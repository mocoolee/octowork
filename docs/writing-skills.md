# 怎麼寫好的 Skill

## 什麼時候該建 Skill

當你發現一件事符合以下任兩項，就值得建成 Skill：

- 每週做 1 次以上
- 每次花 30 分鐘以上
- 流程差不多固定
- 有固定的輸出格式
- 如果做錯，後果明確

## 最簡單的建法

做完一件事之後，直接跟 AI 說：「把剛才的做法存成 Skill。」AI 會自動建一個 SKILL.md。

## SKILL.md 的結構

```markdown
---
name: skill-name
description: "做什麼。觸發詞：xxx、yyy、zzz。"
---

# Skill 名稱

## 執行前
（列出要先讀哪些背景檔）

## 輸入
（使用者會提供什麼）

## 產出格式
（完整的輸出模板）

## 品質規範
（判斷標準、注意事項）
```

## 四個讓 Skill 變好的訣竅

### 1. description 要寫觸發詞

AI 是透過 description 來判斷要不要用這個 Skill。觸發詞寫越多，命中率越高。

不好：`"產出週報。"`
好：`"週報產生器。觸發詞：週報、本週報告、寫週報、交報告、給主管的報告。"`

### 2. 產出格式要完整到可以直接用

不要只寫「產出一份報告」。要用 Markdown 模板展示完整的輸出結構，包含表格、欄位、每個區塊該有什麼內容。

### 3. 加歷史學習

當你發現某個做法特別好或特別差，加到 Skill 裡的「歷史學習」或「品質規範」區塊。這是 Skill 越用越準的原因。

### 4. 指定要讀的背景檔

在「執行前」區塊明確寫「讀 company.md」「讀 products.md」。不寫的話 AI 可能不會自動去讀。

## Skill 的成熟度標籤

每個 Skill 的 frontmatter 有一個 `maturity` 欄位，讓你知道這個 Skill 有多可靠：

| 標籤 | 意思 | 什麼時候改 |
|------|------|-----------|
| `draft` | 剛建的，還在測試 | 自動（剛生成時） |
| `stable` | 用了幾次，產出品質可靠 | 你覺得「這個 Skill 不太需要改了」 |
| `proven` | 用了很多次，千錘百鍊 | 至少用過 10 次，每次都滿意 |
| `deprecated` | 不再使用 | 工作內容改變，或有更好的 Skill 取代 |

跟 AI 說「把 XX Skill 標成 stable」就好，AI 會幫你改 frontmatter。

---

## 什麼時候該淘汰一個 Skill

- 超過 2 個月沒用過
- 每次用都要大幅修改產出才能用
- 有另一個 Skill 做得更好

不要刪掉，把 maturity 改成 `deprecated` 就好。哪天又需要了還找得到。
