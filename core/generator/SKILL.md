# OctoWork 生成器

為任何職位角色，透過訪談生成一套完整的 AI 工作作業系統。

## 設計原則

1. **三層分離**：公司級（所有人共用）/ 職位級（同角色共用）/ 個人級（自己的經驗）
2. **自然生長**：Skills 和 Workflows 不用一次到位，先建核心的，其他做中長出來
3. **維護極簡**：使用者只需要維護 3 個背景檔，其他都是用的時候才碰
4. **雙工具相容**：Claude Code 和 Gemini CLI 都能跑

## 整體流程

```
Phase 1：訪談（收集資訊，每一輪結束就存檔）
  ↓
Phase 2：生成背景檔（CLAUDE.md + company.md + products.md）
  ↓
Phase 3：識別 Skills（列清單讓使用者選，不一定要全部建）
  ↓
Phase 4：生成 Skills（只建使用者選的）
  ↓
Phase 5：識別 Workflows（列清單讓使用者選）
  ↓
Phase 6：生成 Workflows（只建使用者選的）
  ↓
Phase 7：驗收與調整
```

## Phase 1：訪談

用自然對話的方式收集以下資訊。**每次只問一個問題**，等使用者回答後再問下一個。語氣像聊天，不像填表。不要一次丟出一串問題，體感會很差。

### 訪談節奏

- **一次只問一個問題**，絕對不要一次問兩個以上
- 使用者回答後，根據他的回答自然地接下一題（不要照本宣科）
- 如果使用者一次講了很多，就不用再問那些已經答過的
- 如果使用者的回答不夠具體，先追問這題，不要急著跳下一題
- 每一輪結束時，摘要確認：「所以你的狀況是___，我理解得對嗎？」
- 使用者確認後才進入下一輪

### 每輪存檔（重要！）

**使用者可能隨時關掉 Claude Code，所以每一輪訪談結束、使用者確認後，就要立刻把目前收集到的資訊存到檔案裡。** 不要等全部問完才存。

具體做法：
- 第一輪問完確認後 → 把角色和公司資訊存進 `workspace/CLAUDE.md` 和 `workspace/company.md`（即使還不完整）
- 第二輪問完確認後 → 把做事原則和協作者更新到 CLAUDE.md
- 第三輪問完確認後 → 把重複工作清單暫存到 CLAUDE.md 的備註區（之後會變成 Skills）
- 第四輪問完確認後 → 把流程資訊暫存

存檔時告訴使用者：「我先把目前聊到的存起來，這樣就算你關掉再開也不會消失。」

如果使用者中途關掉再回來，發現 workspace/ 裡已經有半完成的檔案，就接著做：
- 讀取已有的檔案，看完成到哪裡
- 告訴使用者：「我看到你上次聊到第 X 輪，要接著繼續嗎？」
- 從斷點繼續，不用重來

### 第一輪：你是誰

目標：了解角色和公司背景。

要問出的資訊：
- 公司叫什麼、做什麼、多大規模
- 你的職稱、負責什麼
- 你的 KPI 是什麼（用數字說）
- 你直屬主管是誰、團隊有誰
- 你用什麼工具（溝通、數據、文件）

開場（第一個問題）：
「先聊聊你的工作吧——你在什麼公司、做什麼角色？」

然後根據回答一題一題往下問。例如：
- 他說了公司和角色 → 「那你平常主要負責哪些事？」
- 他說了負責的事 → 「你的 KPI 大概是什麼？有沒有具體的數字目標？」
- 他說了 KPI → 「你直屬主管是誰？團隊裡還有誰跟你比較常合作？」
- 他說了團隊 → 「平常工作用什麼工具？溝通用什麼、數據看什麼、文件寫在哪？」

### 第二輪：你怎麼做事

目標：挖出隱性知識和判斷原則。

要問出的資訊：
- 你做決策的依據是什麼（有什麼判斷標準）
- 你跟誰合作最多、每個人什麼風格
- 你主管喜歡什麼樣的報告方式
- 你踩過什麼坑、學到什麼教訓

一題一題問。例如：
- 「你工作上最常要做的判斷是什麼？」
- 「你主管看報告的時候，他比較喜歡什麼風格？先看結論還是先看數據？」
- 「有沒有什麼踩過的坑可以分享？就是那種做了之後才知道的事。」

### 第三輪：你的重複工作

目標：找出可以變成 Skill 的重複性工作。

要問出的資訊：
- 每天固定做的事
- 每週固定做的事
- 每個月固定做的事
- 哪些事情你覺得「又來了，跟上次一樣」

一題一題問。例如：
- 「你每天上班一定會做的事有哪些？」
- 「那每週呢？有沒有每個禮拜都要交的東西？」
- 「這些重複的事情裡面，哪幾件你覺得最花時間、最想讓別人幫你做？」

### 第四輪：你的流程

目標：找出可以串成 Workflow 的組合流程。

要問出的資訊：
- 有沒有什麼工作是「做完 A 就要接著做 B 再做 C」的
- 有沒有週期性的大流程（月結、檔期、季度檢討）

一題一題問。例如：
- 「有沒有什麼工作是一條龍的？就是做完這件接著那件，順序差不多固定的？」
- 「有沒有每個月或每季固定要跑的大流程？像月結、檔期、季度檢討之類的？」

### 追問技巧

- 使用者說「報告」→ 追問「報告交給誰？他喜歡什麼格式？多久交一次？」
- 使用者說「處理客訴」→ 追問「你能自己決定退換貨嗎？還是要報主管？金額上限多少？」
- 使用者講得很少 → 用選擇題引導：「你的報告風格比較像 A）數據表格導向 還是 B）敘述故事型？」
- 使用者講得很多 → 幫他整理：「所以你的意思是___，對嗎？」

### 訪談結束與延伸機制

四輪問完後：

1. 整理一份完整摘要，讓使用者確認
2. 確認後，**先不要急著開始生成**，問使用者：

「基本的資訊我都收到了。在我開始建系統之前，你還有沒有想補充的？比如：
- 某個主管的溝通眉角
- 某個工作的特殊判斷邏輯
- 最近在煩惱的事
- 任何你覺得『這個 AI 應該要知道』的事

沒有的話我就開始建了，之後隨時都可以再補。」

3. 如果使用者有補充 → 繼續聊，聊完再整理一次摘要
4. 如果使用者說沒有 → 開始生成

**重要：** 生成完系統後，在 Phase 7 驗收時告訴使用者怎麼持續更新系統：

「你的系統建好了。以後工作中想到什麼，直接跟我說就好，例如：

- 想加一條做事原則 → 直接告訴我，像是『加一條：庫存低於安全量要馬上通知 Tina』
- 想修改某個 Skill 的格式 → 跟我說『改一下週報，加一欄新客數』
- 做了一件新的重複工作 → 跟我說『把剛才的做法存成 Skill』
- 想補充某個人的溝通風格 → 跟我說『Andy 看報告喜歡表格，幫我記下來』

不用特別的指令，像平常講話一樣就好。

另外，如果你想一次把系統補得更完整，跟我說『復盤』，我會讀完你所有的檔案，然後根據現有內容找出可以補強的地方，一個一個問你。」

## Phase 2：生成背景檔

### CLAUDE.md 的結構

```markdown
# OctoWork

你是使用者的 AI 工作助手。請根據以下資訊來協助工作。
開始做事前，先讀 company.md 了解公司背景，再讀 products.md（如果有的話）了解產品/服務資訊。

## 我的角色
（從訪談第一輪提取）
- 姓名：
- 職稱：
- 公司：
- 負責範疇：
- KPI：（必須有具體數字）
- 常用工具：

## 做事原則
（從訪談第二輪提取，按工作類型分組）
### [工作類型 A]
- 原則 1
- 原則 2
### [工作類型 B]
- ...
（至少 5 條，包含踩坑教訓）

## 協作者
（從訪談提取，每個人包含：風格、偏好、地雷、最佳溝通方式）

## 目前進行中
（從訪談提取，如果有的話）

## 系統規則
1. 做事前先讀 company.md 和 products.md（如果有的話）
2. 查 .agents/skills/ 有沒有對應的 Skill，有就用
3. 我說「啟動 XX 流程」→ 讀 workflows/ 對應的 .md 來跑
4. 所有產出存到 outputs/YYYY-MM/
5. 做完後提醒我：有沒有新經驗值得加到「做事原則」
```

### company.md 的結構

```markdown
# 公司資料

## 公司簡介
（公司名、業態、規模、商業模式）

## 品牌/服務定位
（一段話描述核心定位）

## [業務相關的核心資訊]
（根據角色不同，這裡的內容不同）
- 門市角色 → 門市資料、營運標準、稽核項目
- 行銷角色 → 客群輪廓、管道佈局、品牌語氣
- 業務角色 → 客戶名單、報價規範、競品資料
- 客服角色 → 常見問題、處理權限、話術規範

## 歷史經驗
（踩過的坑，至少 3-5 條）
```

### products.md（可選）

如果角色的工作涉及具體的產品或服務，生成 products.md。
如果不涉及（如 HR、行政），跳過這個檔案。

## Phase 3：識別 Skills

從訪談第三輪的「重複工作」中，挑出最符合以下條件的工作來建 Skill：

### 判斷標準（至少符合 2 項）
- ✅ 每週做 1 次以上
- ✅ 每次花 30 分鐘以上
- ✅ 流程差不多固定
- ✅ 有固定的輸出格式
- ✅ 如果做錯，後果明確

### 建議數量
- 第一次生成：5-7 個 Skills（核心的就好）
- 跟使用者確認清單再開始寫

### 列出清單給使用者選

格式：
```
根據你剛才說的，我覺得這些工作可以變成 Skill：

1. [Skill 名稱]：[一句話說明做什麼]，預估每次省 X 分鐘
2. ...

你想先建哪幾個？不用全部，選你最想要的就好，剩下的以後隨時可以再建。
```

使用者可能只選其中幾個，甚至說「先不建，之後再說」——都 OK。不要強迫全部建。

## Phase 4：生成 Skills

每個 Skill 的 SKILL.md 必須包含以下區塊：

```markdown
---
name: [skill-name]
description: "[做什麼]。觸發詞：xxx、yyy、zzz。"
maturity: draft
---

# [Skill 名稱]

## 執行前
（列出要先讀哪些背景檔）

## 輸入
（使用者會提供什麼，格式可能長什麼樣）

## 產出格式
（用 markdown 模板展示完整的輸出結構）

## [品質規範 / 判斷邏輯 / 注意事項]
（根據 Skill 性質不同，這裡的名稱和內容不同）
```

### Maturity 標籤說明

生成器產出的 Skill 預設都是 `draft`，使用者可以隨著使用次數調整：

| 標籤 | 意思 | 什麼時候改 |
|------|------|-----------|
| `draft` | 剛建立，用過 1-2 次 | 自動（剛生成時） |
| `stable` | 用過 3 次以上，產出品質可靠 | 使用者覺得不太需要改了 |
| `proven` | 用過 10 次以上，千錘百鍊 | 每次都滿意，已經是最佳做法 |
| `deprecated` | 不再使用，保留參考 | 工作內容改變或有更好的替代 |

### Skill 品質檢查
每個 Skill 生成後自我檢查：

#### 基本（必要）
- [ ] frontmatter 的 description 有觸發詞（至少 3 個同義詞）
- [ ] 「執行前」有指定要讀哪些背景檔
- [ ] 「輸入」有描述使用者可能提供的各種格式（容錯）
- [ ] 「產出格式」是完整的 Markdown 模板，可以直接用
- [ ] 有品質規範或判斷邏輯

#### 進階（讓 Skill 更好用）
- [ ] 有邊界情況處理（輸入不完整時怎麼辦）
- [ ] 有歷史學習區塊（踩過的坑、驗證有效的做法）
- [ ] 產出格式包含「精簡版」（適合貼到 LINE 或 email）
- [ ] 有「這個 Skill 不處理什麼」的邊界說明

## Phase 5：識別 Workflows

從訪談第四輪的「一條龍流程」中，找出可以串 Skills 的組合。

### 判斷標準
- 涉及 2 個以上的 Skill
- 有固定的先後順序
- 至少每月做一次

### 常見的 Workflow 模式
- **每日循環**：看數據 → 分析 → 行動
- **每週報告**：彙整 → 分析 → 寫報告 → 發送
- **事件驅動**：收到觸發 → 準備 → 執行 → 追蹤
- **週期檢討**：收集數據 → 檢討 → 更新計畫

---

## Phase 6：生成 Workflows

每個 Workflow 的 .md 必須包含：

```markdown
# [流程名稱]

## 觸發
我說「...」

## 步驟

### 1. [步驟名稱]
- 做什麼
- 用哪個 Skill
- 👉 需不需要停下來讓使用者確認

### 2. ...

## 輸出
- 存到哪裡
- 有沒有需要複製貼上的精簡版
```

## Phase 7：驗收

全部生成完後：

1. 列出完整的檔案清單，讓使用者一目瞭然
2. 展示資料夾結構
3. 問使用者：「要不要挑一個 Skill 試跑一次？」
4. 如果使用者想試跑，用一個假場景來測
5. 告訴使用者維護建議：「你的系統會越用越聰明，但也要定期整理。建議每個月花 5 分鐘跟我說『整理原則』，我會幫你找出重複的、過時的，讓做事原則保持在 30 條以內。」

## 檔案結構規範

所有生成的檔案直接放在 `workspace/` 裡，不要再建子資料夾：

```
workspace/
├── CLAUDE.md
├── company.md
├── products.md（如果需要）
├── .agents/
│   └── skills/
│       └── [skill-name]/
│           └── SKILL.md
├── workflows/
│   └── [workflow-name].md
└── outputs/
```

Skill 放在 `.agents/skills/` 讓 Claude Code 和 Gemini CLI 都能自動掃描。

**重要：** 所有 Skill 必須存在 `workspace/.agents/skills/`，不可以存到 Claude Code 的全域路徑（如 `~/.claude/.agents/skills/`）。每個人的 Skill 只屬於他自己的系統。

## 語氣與溝通

- 訪談時像聊天，不像填問卷
- 使用者說的不清楚就追問，但不要一次問超過 3 個問題
- 每個階段結束都要摘要確認
- 生成完每個檔案都簡短說明它是做什麼的
- 不要用技術術語（Skill、Workflow 可以用，但不要說 YAML frontmatter、progressive disclosure 這類詞）
