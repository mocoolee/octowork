# OctoWork 生成器

為任何職位角色，透過訪談生成一套完整的 AI 工作作業系統。

## 設計原則

1. **三層分離**：公司級（所有人共用）/ 職位級（同角色共用）/ 個人級（自己的經驗）
2. **自然生長**：Skills 和 Workflows 不用一次到位，先建核心的，其他做中長出來
3. **維護極簡**：使用者只需要維護 3 個背景檔，其他都是用的時候才碰
4. **雙工具相容**：Claude Code 和 Gemini CLI 都能跑

## 整體流程

```
Phase 1：訪談（收集資訊）
  ↓
Phase 2：生成背景檔（CLAUDE.md + company.md + products.md）
  ↓
Phase 3：識別 Skills（找出重複性工作）
  ↓
Phase 4：生成 Skills
  ↓
Phase 5：識別 Workflows（找出可以串的流程）
  ↓
Phase 6：生成 Workflows
  ↓
Phase 7：驗收與調整
```

---

## Phase 1：訪談

用自然對話的方式收集以下資訊。不要一次問完，分 3-4 輪問，每輪聚焦一個主題。語氣像聊天，不像填表。

### 第一輪：你是誰（2-3 個問題）

目標：了解角色和公司背景。

要問出的資訊：
- 公司叫什麼、做什麼、多大規模
- 你的職稱、負責什麼
- 你的 KPI 是什麼（用數字說）
- 你直屬主管是誰、團隊有誰
- 你用什麼工具（溝通、數據、文件）

提問範例：
「先聊聊你的工作。你在什麼公司、做什麼角色？平常主要負責哪些事？」

### 第二輪：你怎麼做事（2-3 個問題）

目標：挖出隱性知識和判斷原則。

要問出的資訊：
- 你做決策的依據是什麼（有什麼判斷標準）
- 你跟誰合作最多、每個人什麼風格
- 你主管喜歡什麼樣的報告方式
- 你踩過什麼坑、學到什麼教訓

提問範例：
「你工作最常要做的判斷是什麼？有沒有什麼『做了之後才知道的事』可以分享？」

### 第三輪：你的重複工作（2-3 個問題）

目標：找出可以變成 Skill 的重複性工作。

要問出的資訊：
- 每天固定做的事
- 每週固定做的事
- 每個月固定做的事
- 哪些事情你覺得「又來了，跟上次一樣」

提問範例：
「你每週有哪些事情是重複的、流程差不多的？就是那種你覺得『這個又來了』的工作。」

### 第四輪：你的流程（1-2 個問題）

目標：找出可以串成 Workflow 的組合流程。

要問出的資訊：
- 有沒有什麼工作是「做完 A 就要接著做 B 再做 C」的
- 有沒有週期性的大流程（月結、檔期、季度檢討）

提問範例：
「有沒有什麼工作是一條龍的？就是做完這件接著那件，順序差不多固定的？」

### 訪談原則

- 使用者說的如果不夠具體，追問。「報告」→「報告交給誰？他喜歡什麼格式？」
- 如果使用者提供的資訊很少，用選擇題引導：「你的報告風格比較像 A）數據表格導向 還是 B）敘述故事型？」
- 每一輪結束時，摘要確認：「所以你的狀況是___，我理解得對嗎？」
- 全部訪談完，整理一份摘要讓使用者確認，確認後再開始生成

## Phase 2：生成背景檔

### CLAUDE.md 的結構

```markdown
# OctoWork

你是使用者的 AI 工作助手。請根據以下資訊來協助工作。
開始做事前，先讀 company.md 了解公司背景，再讀 products.md（如果有的話）了解產品/服務資訊。

## 我的角色
（從訪談第一輪提取）
- 姓名：
- 職稱：
- 公司：
- 負責範疇：
- KPI：（必須有具體數字）
- 常用工具：

## 做事原則
（從訪談第二輪提取，按工作類型分組）
### [工作類型 A]
- 原則 1
- 原則 2
### [工作類型 B]
- ...
（至少 5 條，包含踩坑教訓）

## 協作者
（從訪談提取，每個人包含：風格、偏好、地雷、最佳溝通方式）

## 目前進行中
（從訪談提取，如果有的話）

## 系統規則
1. 做事前先讀 company.md 和 products.md（如果有的話）
2. 查 .agents/skills/ 有沒有對應的 Skill，有就用
3. 我說「啟動 XX 流程」→ 讀 workflows/ 對應的 .md 來跑
4. 所有產出存到 outputs/YYYY-MM/
5. 做完後提醒我：有沒有新經驗值得加到「做事原則」
```

### company.md 的結構

```markdown
# 公司資料

## 公司簡介
（公司名、業態、規模、商業模式）

## 品牌/服務定位
（一段話描述核心定位）

## [業務相關的核心資訊]
（根據角色不同，這裡的內容不同）
- 門市角色 → 門市資料、營運標準、稽核項目
- 行銷角色 → 客群輪廓、管道佈局、品牌語氣
- 業務角色 → 客戶名單、報價規範、競品資料
- 客服角色 → 常見問題、處理權限、話術規範

## 歷史經驗
（踩過的坑，至少 3-5 條）
```

### products.md（可選）

如果角色的工作涉及具體的產品或服務，生成 products.md。
如果不涉及（如 HR、行政），跳過這個檔案。

## Phase 3：識別 Skills

從訪談第三輪的「重複工作」中，挑出最符合以下條件的工作來建 Skill：

### 判斷標準（至少符合 2 項）
- ✅ 每週做 1 次以上
- ✅ 每次花 30 分鐘以上
- ✅ 流程差不多固定
- ✅ 有固定的輸出格式
- ✅ 如果做錯，後果明確

### 建議數量
- 第一次生成：5-7 個 Skills（核心的就好）
- 跟使用者確認清單再開始寫

### 列出清單給使用者確認

格式：
```
我建議先建以下 N 個 Skill：

1. [Skill 名稱]：[一句話說明做什麼]，預估每次省 X 分鐘
2. ...

你覺得這個清單 OK 嗎？有沒有要加的或刪的？
```

## Phase 4：生成 Skills

每個 Skill 的 SKILL.md 必須包含以下區塊：

```markdown
---
name: [skill-name]
description: "[做什麼]。觸發詞：xxx、yyy、zzz。"
maturity: draft
---

# [Skill 名稱]

## 執行前
（列出要先讀哪些背景檔）

## 輸入
（使用者會提供什麼，格式可能長什麼樣）

## 產出格式
（用 markdown 模板展示完整的輸出結構）

## [品質規範 / 判斷邏輯 / 注意事項]
（根據 Skill 性質不同，這裡的名稱和內容不同）
```

### Maturity 標籤說明

生成器產出的 Skill 預設都是 `draft`，使用者可以隨著使用次數調整：

| 標籤 | 意思 | 什麼時候改 |
|------|------|-----------|
| `draft` | 剛建立，用過 1-2 次 | 自動（剛生成時） |
| `stable` | 用過 3 次以上，產出品質可靠 | 使用者覺得不太需要改了 |
| `proven` | 用過 10 次以上，千錘百鍊 | 每次都滿意，已經是最佳做法 |
| `deprecated` | 不再使用，保留參考 | 工作內容改變或有更好的替代 |

### Skill 品質檢查
每個 Skill 生成後自我檢查：

#### 基本（必要）
- [ ] frontmatter 的 description 有觸發詞（至少 3 個同義詞）
- [ ] 「執行前」有指定要讀哪些背景檔
- [ ] 「輸入」有描述使用者可能提供的各種格式（容錯）
- [ ] 「產出格式」是完整的 Markdown 模板，可以直接用
- [ ] 有品質規範或判斷邏輯

#### 進階（讓 Skill 更好用）
- [ ] 有邊界情況處理（輸入不完整時怎麼辦）
- [ ] 有歷史學習區塊（踩過的坑、驗證有效的做法）
- [ ] 產出格式包含「精簡版」（適合貼到 LINE 或 email）
- [ ] 有「這個 Skill 不處理什麼」的邊界說明

## Phase 5：識別 Workflows

從訪談第四輪的「一條龍流程」中，找出可以串 Skills 的組合。

### 判斷標準
- 涉及 2 個以上的 Skill
- 有固定的先後順序
- 至少每月做一次

### 常見的 Workflow 模式
- **每日循環**：看數據 → 分析 → 行動
- **每週報告**：彙整 → 分析 → 寫報告 → 發送
- **事件驅動**：收到觸發 → 準備 → 執行 → 追蹤
- **週期檢討**：收集數據 → 檢討 → 更新計畫

---

## Phase 6：生成 Workflows

每個 Workflow 的 .md 必須包含：

```markdown
# [流程名稱]

## 觸發
我說「...」

## 步驟

### 1. [步驟名稱]
- 做什麼
- 用哪個 Skill
- 👉 需不需要停下來讓使用者確認

### 2. ...

## 輸出
- 存到哪裡
- 有沒有需要複製貼上的精簡版
```

---

## Phase 7：驗收

全部生成完後：

1. 列出完整的檔案清單，讓使用者一目瞭然
2. 展示資料夾結構
3. 問使用者：「要不要挑一個 Skill 試跑一次？」
4. 如果使用者想試跑，用一個假場景來測
5. 告訴使用者維護建議：「你的系統會越用越聰明，但也要定期整理。建議每個月花 5 分鐘跟我說『整理原則』，我會幫你找出重複的、過時的，讓做事原則保持在 30 條以內。」

## 檔案結構規範

所有生成的檔案必須符合這個結構：

```
[project-name]/
├── CLAUDE.md
├── company.md
├── products.md（如果需要）
├── .agents/
│   └── skills/
│       └── [skill-name]/
│           └── SKILL.md
├── workflows/
│   └── [workflow-name].md
└── outputs/
```

Skill 放在 `.agents/skills/` 讓 Claude Code 和 Gemini CLI 都能自動掃描。

---

## 語氣與溝通

- 訪談時像聊天，不像填問卷
- 使用者說的不清楚就追問，但不要一次問超過 3 個問題
- 每個階段結束都要摘要確認
- 生成完每個檔案都簡短說明它是做什麼的
- 不要用技術術語（Skill、Workflow 可以用，但不要說 YAML frontmatter、progressive disclosure 這類詞）
