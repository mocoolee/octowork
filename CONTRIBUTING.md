# 貢獻指南

歡迎為 OctoWork 貢獻！以下是幾種參與方式。

## 貢獻新的角色範例

這是最有價值的貢獻。每一個新角色範例都會幫助到同職位的人。

### 範例的品質標準

一個合格的範例需要包含：

```
examples/[角色名稱]/
├── CLAUDE.md          ← 完整的角色定義（至少 80 行）
│                         必須包含：角色、KPI、做事原則（至少 5 條）、協作者
├── company.md         ← 真實可信的公司背景（可以是虛構公司，但細節要合理）
├── products.md        ← 如果該角色涉及產品/服務（可選）
├── .agents/skills/    ← 至少 5 個完整的 Skill
│   └── [skill]/
│       └── SKILL.md      每個 Skill 必須有：執行前、輸入、產出格式、品質規範
└── workflows/         ← 至少 2 個 Workflow
    └── [workflow].md     每個 Workflow 必須有：觸發、步驟、輸出
```

### 範例的原則

1. 做事原則要來自實戰經驗，不是通用建議。「週報要簡短」是廢話，「老闆只看三個數字：營收、ROAS、新客比」是經驗。
2. Skill 的產出格式要完整到可以直接用，不是「產出一份報告」。
3. 協作者要有個性——風格、偏好、地雷，不是只有名字和職稱。
4. 歷史經驗至少 3 條，寫踩過的坑和學到的教訓。

### 提交流程

1. Fork 這個 repo
2. 在 `examples/` 裡新增你的角色資料夾
3. 確認符合上面的品質標準
4. 提交 PR，說明這個角色是做什麼的

## 改進生成器

如果你覺得生成器的訪談流程可以更好、模板可以更完整，歡迎修改 `core/generator/` 裡的檔案並提交 PR。

## 改進文件

發現文件有錯字、不清楚、或缺少內容？直接改 `docs/` 裡的檔案並提交 PR。

## 回報問題

在 GitHub Issues 裡開 issue，描述你遇到的問題。如果是生成器產出的品質問題，請附上你的輸入和產出讓我們重現。

## 行為準則

保持友善和建設性。我們歡迎所有背景的貢獻者。
